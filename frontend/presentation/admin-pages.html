🏗️ Architecture Overview

All admin pages follow a consistent pattern: a Server Component layout checks the session for ADMIN role, then a Server Component page fetches data via apiFetch, passes it to Client Components for interactivity, and Server Actions handle mutations with automatic cache revalidation.

Page (Server) → apiFetch (Server) → Backend API → Client Component → Server Action → apiFetch (Action)

🔐 Auth Flow & Layout

Layout Protection

Every admin page is wrapped by admin/layout.tsx, which is a Server Component that calls auth() from NextAuth before rendering anything.

Request → auth() → Session null? → redirect /login
auth() → Role ≠ ADMIN? → redirect /
auth() → AdminSidebar → children (page content)
// admin/layout.tsx — auth guard const session = await auth(); if (!session?.user) { redirect('/login'); } if ((session.user as Record<string, unknown>).role !== 'ADMIN') { redirect('/'); } return ( <div className="flex min-h-screen bg-zinc-50"> <AdminSidebar /> <main className="flex-1 p-8 overflow-auto"> {children} </main> </div> );
🔒 Security
The role check happens server-side — unauthorized users never see admin UI or data. The sidebar only renders for ADMIN users.

🔄 Data Flow Pattern

Every admin page follows the same three-layer architecture:

1. Server Page → apiFetch() → Backend JSON → Render HTML
2. Client Component → User clicks → Server Action → apiFetch() PATCH/DELETE → Backend Response
3. Server Action → revalidatePath() → Page re-renders → Fresh data

apiFetch — The Bridge

apiFetch is a thin wrapper around fetch() that:

  • Prepends BACKEND_URL (default http://localhost:4000)
  • Reads the authjs.session-token cookie and sends it as Authorization: Bearer <token>
  • Sets Content-Type: application/json when body is present
  • Works in both Server Components and Server Actions
// From frontend/src/lib/api.ts — simplified export async function apiFetch(path: string, options: ApiOptions = {}) { const { auth = true, ...fetchOptions } = options; const headers = new Headers(fetchOptions.headers); if (auth) { const token = await getAuthToken(); // reads cookie if (token) headers.set('Authorization', `Bearer ${token}`); } return fetch(`${BACKEND_URL}${path}`, { ...fetchOptions, headers }); }

🔍 Search & Pagination

Pattern

Both Users and Courses pages use URL-based search params for search and pagination. This means:

  • Search is a plain HTML <form> — works without JavaScript
  • Pagination uses <a> links — works without JavaScript
  • The page reads params from searchParams prop (awaited)
  • No client-side state needed for pagination
Search Form
MethodGET (form default)
Input namesearch
Default valuefrom searchParams.search
Pagination Links
Params?page=N&search=...
Previoushidden on page 1
Nexthidden on last page
// Search form — submits via GET, preserving search param across pages <form className="relative mb-6 max-w-md"> <Search className="absolute left-3 top-1/2 -translate-y-1/2 w-4 h-4 text-zinc-400" /> <input name="search" defaultValue={search} placeholder="Search..." /> </form> // Pagination — preserves search param across pages <a href={`/admin/users?page=${page - 1}${search ? `&search=${search}` : ""}`}> Previous </a>
💡 Server-Side Rendering
Because search and pagination use URL search params, every page load fetches fresh data from the backend. The search input's defaultValue is set from the current search param, so it reflects the active search.

Shared dependencies:
• apiFetch from @/lib/api — all data fetching

• sonner Toaster — toast notifications (root layout)
• lucide-react — icons (sidebar, buttons, cards)
• Tailwind CSS — all styling (no CSS modules)

📌 Note
All Server Actions use revalidatePath() after a mutation. This tells Next.js to re-fetch the page data on the next request, so the UI always reflects the latest database state. There is no optimistic UI — the page re-renders with fresh data from the backend.

Key Architectural Decisions

DecisionRationale
Auth check in layout, not per-page Single responsibility — layout guards all admin routes
Server Components for data fetching No client-side loading states, direct backend access, no waterfall
Client Components only for interactivity Minimal JS shipped to browser — just forms and buttons
Server Actions for mutations Type-safe, automatically revalidate cache, work with forms
URL params for search + pagination Works without JS, bookmarkable, shareable URLs
Self-operation guards in client + server Backend enforces it (can't change own role), frontend also disables the UI
sonner for toast notifications Lightweight, already in package.json, richColors look clean
